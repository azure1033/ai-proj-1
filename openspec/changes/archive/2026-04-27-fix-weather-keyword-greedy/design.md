## Context

`classify_intent(query)` 在 `/ask` 路由中作为意图分类的核心函数（在 `classify_complexity` 判定为简单查询后调用）。当前实现使用单关键词匹配：只要 query 包含 weather_keywords 列表中的任意一个词，直接返回"天气查询"。

问题：列表中的某些词太泛（"今天"、"明天"、"穿"、"这里"），导致非天气查询被误判。

## Goals / Non-Goals

**Goals:**
- "今天" 单独出现不再触发天气意图
- "今天天气/明天会下雨/这里热不热" 等真正天气查询仍然正确触发
- 误判的查询能走到正确的意图分支（LLM 分类或 Agent）

**Non-Goals:**
- 不改变 LLM 分类逻辑
- 不改变 Agent 路由逻辑
- 不改变天气工具行为

## Decisions

### Decision 1: 天气关键词分层

**选择**: 将 weather_keywords 拆分为 primary（主）和 secondary（辅助）

```
主关键词（命中任意一个即可视为天气）:
  天气, 天气如何, 天气怎么样, 温度, 多少度, 热, 冷, 热不热, 冷不冷,
  下雨, 会下雨, 下雨吗, 下雪, 会下雪, 风, 风力, 刮风, 有没有风, 风大吗,
  湿度, 潮湿, 干燥, 空气质量, AQI, 口罩, 污染, 空气怎么样,
  紫外线, 阳光, 太阳, 雾霾, 雾, 能见度

辅助关键词（必须搭配主关键词才生效）:
  今天, 明天, 后天, 这里, 这个城市, 当地, 穿, 衣服, 衣物, 外套, 羽绒,
  需要穿, 出行, 出去, 户外, 运动, 能不能出去
```

**判定逻辑**:
```python
# 1. 辅助关键词不足 → 不是天气，委托 LLM
if not any(kw in query for kw in weather_primary):
    return classify_by_llm(query)

# 2. 主关键词命中 → 天气查询
return "天气查询"
```

**理由**:
- "今天/明天" 等词在真正天气查询中总是和主关键词一起出现（"今天天气"、"明天会下雨吗"）
- 单独出现时（"今天有什么新闻"）不应触发天气
- 最小改动，不引入额外复杂度

### Decision 2: 保留辅助关键词的语义价值

虽然辅助关键词不再单独触发天气，但它们仍出现在 weather_keywords 的场景中，保持代码可读性。主关键词本身已经足够覆盖所有天气场景。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 某些单字天气查询被误判为非天气 | "热"和"冷"在主关键词中，单独出现仍触发天气 |
| "这里天气" 等查询仍正常 | "天气" 是主关键词 ✓ |
| 用户只说 "明天呢"（上下文对话中） | 这是对话上下文问题，非本 fix 范围 |
